**Почему именно ОЯТ — это умный выбор для хакатона**

Во-первых, это глобально нерешённая проблема. В мире накоплено около 400 000 тонн ОЯТ, и большинство стран до сих пор не имеют окончательного решения по захоронению. Финляндия строит первое в мире постоянное хранилище Onkalo — оно откроется только в 2030-х. Это значит, что жюри из отрасли воспримет тему серьёзно.

Во-вторых, задача хорошо декомпозируется на ваши три роли и не требует реальных секретных данных — физические параметры ОЯТ публично описаны в литературе МАГАТЭ.



Они естественно дополняют друг друга и делятся по ролям без пересечений:

ML/Data Science пишет физические модели: формулу остаточного тепловыделения (она есть в стандарте ANS-5.1) и расчёт активности по изотопам через закон распада. Никакого обучения модели — чистая физика, что полностью снимает вопрос галлюцинаций.

Backend делает FastAPI с двумя эндпоинтами: расчёт тепловыделения сборки по параметрам и алгоритм размещения в хранилище. Плюс генератор синтетического парка сборок для демо.

Frontend рисует два экрана. Первый — интерактивная сетка хранилища, где ячейки окрашены по температуре (тепловая карта), можно кликнуть на сборку и увидеть её паспорт. Второй — временной график активности по изотопам с логарифмической шкалой времени (от 1 года до 10 000 лет).

**Риски и как их закрыть**

Единственный реальный риск — физические формулы. Формула остаточного тепловыделения выглядит громоздко, но для демо достаточно упрощённой версии и честно сказать жюри "мы используем приближение ANS-5.1". Это воспримут как плюс, а не минус — покажет, что вы знаете источники.


### Физическое ядро — что считаем

**Остаточное тепловыделение сборки**

Используем упрощённую формулу Вэя-Вигнера (Way-Wigner approximation) — она даёт погрешность ~20% но достаточна для демо:

```
P(t) = P₀ · 0.066 · (t⁻⁰·² - (t + T)⁻⁰·²)
```

где P₀ — мощность реактора на сборку (МВт), t — время после остановки (секунды), T — время работы в реакторе (секунды).

На входе у нас три параметра сборки: начальное обогащение (%), глубина выгорания (МВт·сут/тУ), возраст после выгрузки (лет). Из выгорания и времени работы получаем P₀ и T.

**Активность по ключевым изотопам**

Для каждого изотопа закон простой:

```
A(t) = A₀ · e^(-λt),  где λ = ln(2) / T½
```

Берём пять изотопов, которые реально определяют опасность на разных горизонтах времени:

|Изотоп|Период полураспада|Доминирует|
|---|---|---|
|Cs-137|30 лет|10–300 лет|
|Sr-90|29 лет|10–300 лет|
|Am-241|432 года|100–1000 лет|
|Pu-239|24 100 лет|1000+ лет|
|Tc-99|213 000 лет|очень долго|

Начальные активности A₀ берём из публичных таблиц МАГАТЭ для типового топлива ВВЭР-1000 с выгоранием 40–50 МВт·сут/тУ, масштабируем на фактическое выгорание линейно (это грубо, но честно скажем жюри).

### Алгоритм размещения в хранилище

Хранилище — сетка N×M ячеек. Каждая сборка имеет тепловыделение P(t) на момент размещения. Ограничение: сумма тепловыделений соседей ячейки (4 соседа по сторонам) не должна превышать порог Q_max (например, 2 кВт суммарно).

**Алгоритм — жадная укладка с сортировкой:**

Шаг 1. Считаем P(t) для каждой сборки в парке.

Шаг 2. Сортируем сборки по убыванию тепловыделения — самые горячие размещаем первыми, потому что для них труднее найти место.

Шаг 3. Для каждой сборки перебираем ячейки сетки в случайном порядке (shuffle, чтобы не было артефактов края). Выбираем первую ячейку, где тепловой constraint выполнен.

Шаг 4. Если сборка не помещается ни в одну ячейку — помечаем как "требует выдержки", то есть нужно подождать пока остынет.

Шаг 5. На выходе — матрица размещения и список непомещённых сборок с датой, когда они остынут достаточно.

Это не оптимальный алгоритм в математическом смысле, но для демо за 24 часа — честный и объяснимый. При желании можно добавить второй проход с локальным обменом ячеек (swap) для улучшения плотности.

### План по ролям и времени

**Часы 1–3: Общий старт**

Все вместе. Договариваетесь о структурах данных — это критично. Фиксируете JSON-схему сборки:

json

```json
{
  "id": "FA-001",
  "enrichment": 4.5,
  "burnup": 45000,
  "discharge_year": 2019,
  "reactor_operation_years": 4
}
```

И схему ответа размещения:

json

```json
{
  "grid": [[null, "FA-001", "FA-003"], ...],
  "unplaced": ["FA-007"],
  "unplaced_ready_year": {"FA-007": 2026}
}
```

Пока договариваетесь — ML сразу пишет физическое ядро, это самая рисковая часть.

**Часы 3–10: Параллельная разработка**

ML/Data Science пишет physics.py — функции heat_output(assembly, t) и decay_activity(assembly, t, isotopes). Затем optimizer.py — алгоритм жадной укладки. Покрывает unit-тестами на синтетических данных — это важно, физику легко сломать.

Backend делает FastAPI: три эндпоинта. POST /assemblies/calculate — принимает список сборок, возвращает тепловыделения. POST /storage/optimize — запускает алгоритм размещения. GET /assemblies/{id}/decay — возвращает кривую активности по времени для графика.

Frontend начинает с двух компонентов-заглушек с хардкоженными данными — не ждёт бэкенд. Компонент StorageGrid (сетка хранилища, тепловая карта) и компонент DecayChart (график активности).

**Часы 10–16: Интеграция**

Frontend подключается к реальному API. Backend интегрирует physics core. Главный риск здесь — несовпадение форматов данных, поэтому схемы из часа 1 критически важны.

**Часы 16–21: Синтетические данные и полировка**

ML генерирует реалистичный парк из 50–100 сборок с разными параметрами — это ваше демо. Frontend полирует UI: цветовая шкала тепловой карты, тултипы при клике на ячейку, переключатель временной шкалы на графике.

**Часы 21–24: Презентация**

Готовите нарратив. Главный тезис для жюри: "Мы не делаем ИИ который управляет хранилищем. Мы делаем инструмент который помогает инженеру принять обоснованное решение с полным объяснением каждого шага." Это снимает все вопросы про надёжность и галлюцинации сразу.
